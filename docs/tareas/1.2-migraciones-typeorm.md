# Tarea 1.2 — Migraciones TypeORM y eliminación de `synchronize`

> Fase 1 — Integridad de datos · 🔴 Ítem 2 del plan · Depende de: 1.1 · Reglas: [README.md](README.md)

## Contexto / problema

`src/app.module.ts` usa `synchronize: true`: TypeORM altera el esquema automáticamente al arrancar. Con datos reales del cliente esto puede **borrar o alterar columnas** ante cualquier cambio de entidad. Es el riesgo más grave del proyecto.

Patrón objetivo: el esquema evoluciona **solo** por migraciones versionadas e inmutables (ver Patrones grupo B del plan maestro).

## Archivos

- `src/config/data-source.ts` (nuevo — DataSource para el CLI de TypeORM)
- `src/migrations/` (carpeta nueva)
- `src/app.module.ts`
- `package.json`
- `README.md` del backend

## Pasos

1. Crear `src/config/data-source.ts`: exporta un `new DataSource({...})` que lea `.env` con `dotenv/config`, con `entities: ['src/**/*.entity.ts']` y `migrations: ['src/migrations/*.ts']`.
2. Scripts en `package.json`:
   ```json
   "typeorm": "typeorm-ts-node-commonjs -d src/config/data-source.ts",
   "migration:generate": "pnpm typeorm migration:generate",
   "migration:run": "pnpm typeorm migration:run",
   "migration:revert": "pnpm typeorm migration:revert"
   ```
3. Con la BD local ya sincronizada por el `synchronize` actual, generar la migración inicial: `pnpm migration:generate src/migrations/InitialSchema`.
4. **Revisar la migración a mano**: debe crear todas las tablas del dominio — verificar contra los decoradores `@Entity()` reales (vales, vale_tallas, produccion_regs, referencias, tarifas, receta_items, materiales, operarios, pagos, ventas; los nombres exactos salen de las entidades).
5. En `app.module.ts`: `synchronize: false` (sin condición, jamás condicionado a env) y `migrationsRun: false`.
6. Probar desde cero: `docker compose down -v && docker compose up -d`, luego `pnpm migration:run` → esquema completo creado; la app arranca.
7. Documentar en el README del backend el flujo: cambiar entidad → `migration:generate` → revisar el SQL generado → `migration:run`. Regla: una migración aplicada en cualquier entorno compartido es inmutable; los cambios van en una migración nueva.

## Criterio de aceptación

- [ ] BD recreada desde cero **solo** con `pnpm migration:run`.
- [ ] La app arranca con `synchronize: false` y `GET /vales` responde (vacío es válido; el seed llega en 1.3).
- [ ] Commit: `feat(db): typeorm migrations, synchronize disabled permanently`.
