# Tarea 5.2 — Consistencia de dominio (enums, fechas, dinero)

> Fase 5 — Pulido · 🟡 Ítem 13 del plan · Depende de: 5.1 · Reglas: [README.md](README.md)

## Contexto

Tres inconsistencias de modelado:

1. `Pago.etapa` es string libre (el seed guarda `'Cortador'`) mientras producción usa el enum `Oficio` — mismo concepto, dos representaciones.
2. `fecha` es `string` en todas las entidades (columna varchar) y los DTOs solo validan `@IsString` — acepta `"hola"` como fecha.
3. El dinero debe estar en `numeric` de PostgreSQL con el `decimal.transformer` existente (`src/common/transformers/decimal.transformer.ts`) en **todas** las columnas de montos — verificar una por una (`precio`, `precioVenta`, `valor` de tarifas, `montoPagado`, `monto`, `precioUnitario`).

## Archivos

- Entidades afectadas + migraciones correspondientes
- DTOs con campos `fecha` (`@IsDateString`) 
- `src/seed/seed.service.ts` (usar el enum)

## Pasos

1. `Pago.etapa` → tipo enum `Oficio` (columna enum o varchar con check; migración con `USING` para convertir datos existentes — los valores actuales ya coinciden con el enum).
2. Columnas `fecha` → tipo `date` (migración `ALTER ... TYPE date USING fecha::date`; verificar antes que todos los valores existentes son fechas válidas). En los DTOs: `@IsDateString({}, { message: 'La fecha debe tener formato YYYY-MM-DD' })`.
3. Auditar columnas de dinero: cada una con `type: 'numeric', precision: 12, scale: 2, transformer: decimalTransformer` (ajustar precisión a la real). Migración si alguna está como `int`/`float`. Documentar la política de redondeo en un comentario del transformer (p. ej. `Math.round` a 2 decimales).
4. **Bug de zona horaria**: las fechas "de hoy" se generan con `new Date().toISOString().split('T')[0]` (UTC) en `pagos.service.ts` (~31 y ~72), `vales.controller.ts` (~29) y `ventas.service.ts` (~46) — en Colombia (UTC-5), después de las 7 p.m. el pago/venta queda fechado al **día siguiente**. Crear un helper único `hoyLocal()` en `src/common/` que devuelva `YYYY-MM-DD` en la zona horaria del negocio (`TZ` por env, default `America/Bogota`, vía `Intl.DateTimeFormat('en-CA', { timeZone })`) y usarlo en los tres puntos. Test que lo fije.
4. Correr las migraciones sobre una copia con los datos del seed para verificar que no hay pérdida.

## Criterio de aceptación

- [ ] Migraciones corren sobre una BD con datos existentes sin pérdida ni error.
- [ ] `POST` con `fecha: "no-es-fecha"` → 400 con mensaje en español.
- [ ] Un monto con decimales (p. ej. tarifa 1200.50) persiste y se lee exacto.
- [ ] Tests y frontend siguen verdes.
- [ ] Commit: `refactor(domain): oficio enum in pagos, date columns, numeric money audit`.
