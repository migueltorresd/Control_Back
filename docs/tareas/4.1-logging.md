# Tarea 4.1 — Logging estructurado

> Fase 4 — Operabilidad · 🟡 Ítem 11 del plan · OWASP A09 · Depende de: fase 2 · Reglas: [README.md](README.md)

## Contexto

Hoy solo hay `console.log` sueltos (seed) y ningún registro de requests. Para soporte post-entrega se necesita: traza de cada request y rastro explícito de las operaciones de dinero.

## Archivos

- `src/common/interceptors/logging.interceptor.ts` (nuevo)
- `src/app.module.ts` (registrar interceptor global vía `APP_INTERCEPTOR`)
- `src/modules/pagos/pagos.service.ts`, `src/modules/vales/produccion.service.ts` (logs de negocio)
- Reemplazos de `console.log` restantes

## Pasos

1. Crear `LoggingInterceptor` (patrón Interceptor — cross-cutting fuera de los services): por request loguea método, ruta, status, duración en ms y `username` del JWT si existe (`request.user`). Nivel `log` para 2xx/3xx, `warn` para 4xx, `error` para 5xx.
2. Registrarlo global con `APP_INTERCEPTOR`.
3. Reemplazar todo `console.log`/`console.error` del código por `Logger` de Nest con contexto de clase (`new Logger(PagosService.name)`).
4. Logs de negocio explícitos (quién-qué-cuánto):
   - Pago creado: regId, valeId, operarioId, monto, usuario que ejecuta.
   - Pago anulado: ídem.
   - Producción aprobada/revertida: regId, monto congelado.
5. Usar el `Logger` **nativo** de NestJS — no instalar pino ni winston (suficiente a esta escala; cambiar de librería no es parte de esta tarea).

## Criterio de aceptación

- [x] Cada request deja una línea con método, ruta, status, ms y usuario.
- [x] Un pago y su anulación dejan rastro identificable en el log.
- [x] `grep -rn "console\." src/` → sin resultados (fuera de scripts de seed/create-admin, donde es aceptable).
- [x] Commit: `feat(observability): request logging interceptor and business operation logs`.
