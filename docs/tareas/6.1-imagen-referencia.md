# Tarea 6.1 — Imagen del modelo (referencia): backend

> Fase 6 — Funcionalidades · Feature solicitada por el cliente · Depende de: 1.2 (migraciones) y 2.2 (auth) · Reglas: [README.md](README.md)
> Contraparte frontend: `e:\proyectos\control-produccion\docs\tareas\F6-imagen-referencia.md`

## Contexto

Cada referencia (modelo de zapato) debe poder tener **una** foto del producto, que el frontend mostrará en los vales para identificar visualmente qué se está fabricando. La sube el administrador al crear/editar el modelo (desde el celular puede ser foto de cámara).

**Decisiones de arquitectura (no reabrir):**
- Almacenamiento: **disco local** en `uploads/referencias/` (volumen en compose). A esta escala no se justifica S3/object storage; disparador documentado: si algún día hay múltiples instancias del backend → migrar a object storage.
- Una imagen por referencia; re-subir **reemplaza** la anterior.
- El GET de la imagen es **público** (`@Public()`): son fotos de zapatos (no sensibles) y los tags `<img>` no envían headers de autorización. El POST es solo ADMIN.

## Seguridad del upload (OWASP A03/A05 — obligatorio)

- Nunca usar el nombre de archivo del cliente: el nombre lo genera el servidor (`<refId>.<ext>`).
- Whitelist de tipos: solo `image/jpeg`, `image/png`, `image/webp` (validar mimetype **y** extensión derivada del mimetype, no del filename).
- Tamaño máximo: 5 MB.
- Servir siempre con `Content-Type` correcto y `X-Content-Type-Options: nosniff` (helmet ya lo pone global).
- `uploads/` va en `.gitignore` y `.dockerignore`.

## Archivos

- `src/modules/referencias/entities/referencia.entity.ts` + nueva migración (columna `imagenExt` varchar nullable)
- `src/modules/referencias/referencias.controller.ts`, `referencias.service.ts`, `referencias.repository.ts`
- `src/config/env.validation.ts`, `.env.example` (variable `UPLOADS_DIR`, default `./uploads`)
- `compose.yaml` (volumen para uploads — si la tarea 4.6 ya se hizo; si no, anotarlo allí)

## Pasos

1. `pnpm add -D @types/multer` (multer ya viene con `@nestjs/platform-express`).
2. Migración: columna `imagenExt` (varchar corto, nullable) en `referencias` — guarda solo la extensión (`jpg`|`png`|`webp`); la ruta se deriva (`UPLOADS_DIR/referencias/<id>.<ext>`).
3. `POST /api/v1/referencias/:id/imagen` (`@Roles(ADMIN)`): `FileInterceptor('imagen')` + `ParseFilePipe` con `MaxFileSizeValidator(5MB)` y `FileTypeValidator(/^image\/(jpeg|png|webp)$/)`. El service: valida que la referencia exista, borra la imagen anterior si la extensión cambia, escribe el archivo con nombre generado, actualiza `imagenExt`. Errores en español.
4. `GET /api/v1/referencias/:id/imagen` (`@Public()`): si la referencia no tiene imagen → 404; si tiene → stream del archivo con `Content-Type` correcto y `Cache-Control: public, max-age=86400`.
5. `DELETE /api/v1/referencias/:id/imagen` (`@Roles(ADMIN)`): borra archivo y limpia `imagenExt`.
6. Incluir `tieneImagen: boolean` (derivado de `imagenExt`) en las respuestas de `GET /referencias` para que el frontend sepa si pedir la imagen.
7. Crear el directorio al arrancar si no existe (en el service, `mkdir recursive`). Añadir `uploads/` a `.gitignore` y `.dockerignore`.

## Criterio de aceptación

- [ ] Subir un JPG de 2 MB como ADMIN → 201; `GET .../imagen` lo devuelve con `Content-Type: image/jpeg` **sin token**.
- [ ] Subir un `.exe` renombrado a `.jpg` (mimetype real no-imagen) → 400 (multer valida el mimetype reportado; con `FileTypeValidator` de Nest ≥10 verificar que valida magic bytes — si la versión instalada no lo hace, validar la firma del archivo manualmente: primeros bytes JPEG/PNG/WEBP).
- [ ] Archivo de 6 MB → 400 con mensaje en español.
- [ ] Re-subir reemplaza la imagen; el archivo viejo no queda huérfano en disco.
- [ ] Sin imagen → `GET .../imagen` 404 y `tieneImagen: false` en el listado.
- [ ] Commit: `feat(referencias): product image upload with strict validation and public serving`.
