# Tarea 2.2 — Módulo de autenticación JWT con roles

> Fase 2 — Seguridad perimetral · 🔴 Ítem 1 del plan · OWASP A01/A02/A07 · Depende de: 2.1 · Reglas: [README.md](README.md)
> Diseño completo y justificación: sección **1.1** del [plan maestro](../PLAN-MEJORAS-CALIDAD.md)

## Contexto / problema

La API está completamente abierta: cualquiera con acceso a la red puede pagar, anular pagos y borrar registros. Decisión tomada: **JWT autogestionado** (sin Auth0/Cognito — empresa de 10 personas), con roles desde el día uno aunque hoy solo exista el administrador, para que dar acceso a operarios a futuro no requiera re-arquitectura.

## Pasos

1. `pnpm add @nestjs/jwt @nestjs/passport passport passport-jwt bcrypt` y `pnpm add -D @types/passport-jwt @types/bcrypt`.
2. Entidad `Usuario`: `id` uuid PK, `username` único, `passwordHash`, `rol` enum (`ADMIN` | `OPERARIO`), `operarioId` nullable FK a `operarios` (preparado para el acceso futuro de operarios), `activo` boolean default true. Generar migración.
3. Variables nuevas (Joi requeridas + `.env.example`): `JWT_SECRET` (string, mínimo 32 chars — Joi `min(32)`), `JWT_EXPIRES_IN` (default `8h`).
4. `POST /auth/login` (público, DTO con `username`/`password` requeridos): busca usuario activo, `bcrypt.compare`; si falla → 401 `'Credenciales inválidas'` (mismo mensaje exista o no el usuario — no revelar cuál falló). Respuesta: `{ accessToken, usuario: { username, rol } }`. Claims del JWT: `sub` (id), `username`, `rol`, `operarioId`.
5. `JwtStrategy` (passport-jwt, Bearer header, secret de ConfigService).
6. Guards globales en `AppModule` vía `APP_GUARD` (orden: `JwtAuthGuard`, luego `RolesGuard`) — **fail-closed**: todo endpoint exige token salvo lo marcado `@Public()`. `RolesGuard` lee metadata de `@Roles(...)`; sin metadata de roles, basta estar autenticado.
7. Marcar **todos** los controllers existentes (materiales, operarios, referencias, vales, pagos, ventas) con `@Roles(Rol.ADMIN)` a nivel de clase. Solo `/auth/login` lleva `@Public()` (y `/health` cuando exista, tarea 4.4).
8. `PATCH /auth/password` (autenticado, cualquier rol): DTO exige contraseña actual + nueva; política: mínimo 10 caracteres y distinta del username. Verifica la actual con bcrypt antes de cambiar.
9. Script `pnpm create-admin` (mismo patrón standalone que el seed de la tarea 1.3): crea el admin leyendo `ADMIN_USERNAME`/`ADMIN_PASSWORD` del env (misma política de contraseña); falla con mensaje claro si ya existe. Estas dos variables **no** van al esquema Joi de la app (solo las usa el script).
10. Logging de autenticación (OWASP A07/A09): `Logger.warn` en cada login fallido (username e IP) y `Logger.log` en cada login exitoso.

## Criterio de aceptación

- [x] Sin token: `GET /vales` → 401. Con token de login válido → 200. Token alterado o expirado → 401.
- [x] `pnpm create-admin` crea el usuario; segundo intento falla informando que ya existe.
- [x] Cambio de contraseña funciona; la contraseña anterior deja de servir en el siguiente login.
- [x] Logins fallidos visibles en el log del servidor.
- [x] Commit: `feat(auth): jwt authentication with role-based guards (fail-closed)`.
