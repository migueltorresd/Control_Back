# Tarea 5.1 — Mappers tipados y fin de los `any`

> Fase 5 — Pulido · 🟡 Ítem 13 del plan · Depende de: fase 4 · Reglas: [README.md](README.md)

## Contexto

Quedan `any` que debilitan el tipado: `mapToFrontend(v: any)` en `vales.controller.ts:78`, los `as any` del seed, `exception: any`/`errors: any` en el filtro. Patrón objetivo: **Mapper/Presenter formal** — las entidades nunca salen por HTTP, cada audiencia tiene sus DTOs de respuesta (grupo B del plan maestro; es la base del futuro storefront).

## Archivos

- `src/modules/vales/presenters/vale.presenter.ts` (nuevo) + interfaces de respuesta
- `src/modules/vales/vales.controller.ts`
- `src/seed/seed.service.ts`
- `src/common/filters/http-exception.filter.ts`
- `tsconfig.json`

## Pasos

1. Extraer `mapToFrontend` a un presenter tipado: input `Vale` (entidad), output interface `ValeResponse` explícita (el shape actual exacto — no cambiar el contrato del frontend). Aplicar el mismo criterio a cualquier otro mapeo inline.
2. Seed: eliminar los `as any` creando los objetos anidados (tarifas, receta, tallas) con los tipos reales de las entidades (`DeepPartial<T>` o constructores tipados).
3. Filtro de excepciones: tipar `exception: unknown` con narrowing (`instanceof`), eliminar `any`.
4. `tsconfig.json`: revisar y activar las strict flags que falten (`strict: true` idealmente — incluye `noImplicitAny`, `strictNullChecks`); corregir todo lo que destape. Si `strictNullChecks` destapa demasiado, activarlo igual y corregir: es donde viven los bugs.

## Criterio de aceptación

- [ ] Sin `any` injustificados en `src/` (los justificados llevan comentario del porqué).
- [ ] Build con strict flags verde; tests verdes; el frontend funciona sin cambios (shapes idénticos).
- [ ] Commit: `refactor(types): typed presenters, strict tsconfig, no implicit any`.
