# Tarea 4.2 — Auditoría mínima (audit trail)

> Fase 4 — Operabilidad · 🟡 Ítem 11 / patrón Audit Trail · OWASP A09 · Depende de: 4.1 · Reglas: [README.md](README.md)

## Contexto

Cuando el cliente pregunte "¿quién aprobó/pagó esto y cuándo?", el log no basta (rota, se pierde). Se necesita una tabla de auditoría **escrita dentro de la misma transacción** que la operación — sin event sourcing ni infraestructura extra (decisión documentada en Patrones grupo B del plan maestro).

## Archivos

- Nuevo módulo ligero `src/modules/auditoria/`: entidad, repository, service, controller (solo lectura), módulo
- Nueva migración (tabla `auditoria`)
- `src/modules/pagos/pagos.service.ts`, `src/modules/vales/produccion.service.ts` (puntos de registro)

## Pasos

1. Entidad `Auditoria`: `id` (uuid), `fecha` (timestamp default now), `usuario` (username del JWT), `accion` (string: `PAGAR`, `ANULAR_PAGO`, `APROBAR_PRODUCCION`, `REVERTIR_PRODUCCION`, `ELIMINAR_REGISTRO`), `entidad` (string), `entidadId` (string), `detalle` (jsonb: estado anterior/nuevo, montos, ids relacionados). Migración correspondiente.
2. `AuditoriaService.registrar(entry, manager)`: **siempre recibe el EntityManager de la transacción en curso** — si la operación hace rollback, la auditoría también.
3. Integrar en: pagar (individual y lote), anular pago, aprobar producción, revertir producción, eliminar registro. El `usuario` viene del request (propagarlo desde el controller al service como parámetro — no usar estado global).
4. `GET /api/v1/auditoria` solo `@Roles(ADMIN)`, paginado (`page`/`limit`), orden descendente por fecha, filtro opcional por `entidadId`.

## Criterio de aceptación

- [ ] Pagar y luego anular un registro deja **dos filas** con acción, usuario y detalle correctos.
- [ ] Si la transacción de pago falla (p. ej. estado inválido), no queda fila de auditoría huérfana.
- [ ] `GET /auditoria` responde paginado solo para ADMIN.
- [ ] Commit: `feat(auditoria): transactional audit trail for money and approval operations`.
