# Tarea 1.4 — IDs por secuencia de PostgreSQL

> Fase 1 — Integridad de datos · 🟠 Ítem 5 del plan · Depende de: 1.2 · Reglas: [README.md](README.md)

## Contexto / problema

Los IDs de presentación (`V-0001`, `VT-0001`, `PG-...`) se generan con *read-then-write*:

- `ValesService.create()` (`vales.service.ts:39-47`): lee el último vale, parsea el número y suma 1. Dos requests simultáneos generan el **mismo ID** → violación de PK. Además, si `last.id` está malformado, `parseInt` da `NaN` → ID `V-0NaN` (sin guardia).
- `VentasService.create()` (`ventas.service.ts:32-43`): mismo patrón.
- `PagosService` usa `PG-${Date.now()}-...`: puede colisionar en lotes concurrentes.

Patrón objetivo: **la BD es la única autoridad de identidad** (secuencias). Esto además habilita correr múltiples instancias del backend (statelessness — ver visión arquitectónica del plan maestro).

## Archivos

- Nueva migración en `src/migrations/`
- `src/modules/vales/vales.repository.ts` y `vales.service.ts`
- `src/modules/ventas/ventas.repository.ts` y `ventas.service.ts`
- `src/modules/pagos/pagos.repository.ts` y `pagos.service.ts`

## Pasos

1. Migración: crear `vales_seq`, `ventas_seq`, `pagos_seq` e inicializarlas con el máximo actual extraído de los IDs existentes, p. ej.:
   ```sql
   CREATE SEQUENCE IF NOT EXISTS vales_seq;
   SELECT setval('vales_seq', COALESCE((SELECT MAX(substring(vale from 3)::int) FROM vales), 0));
   ```
   (ajustar nombres de tabla/columna a los reales; equivalente para ventas con `substring(id from 4)` y pagos).
2. En cada repository, método `nextId(manager?: EntityManager)` que ejecute `SELECT nextval('vales_seq')` **con el EntityManager de la transacción activa** (query parametrizada/estática, sin interpolación de input) y formatee: `V-` + `String(n).padStart(4, '0')` (ventas `VT-`, pagos `PG-`).
3. Eliminar `findLast()` y todo el parseo de IDs en `vales.service.ts` y `ventas.service.ts` (incluido el `parseInt` sin guardia).
4. Pagos: reemplazar la generación `PG-${Date.now()}-...` (individual y lote) por la secuencia, obtenida dentro de la transacción existente.
5. La creación del vale (`crearConRelaciones`) obtiene el ID dentro de la **misma transacción** que inserta vale + tallas.

## Criterio de aceptación

- [x] Script o test que dispara 10 `POST /vales` en paralelo (`Promise.all`) → 10 vales con IDs únicos consecutivos, cero errores de PK.
- [x] Mismo criterio para ventas y para pagos en lote.
- [x] Commit: `fix(ids): postgres sequences for vale/venta/pago ids, removes race condition`.

## Registro de verificación

**2026-06-11 · Auditada y aprobada ✅** (commit `f5f2714` + fix de auditoría)

- **Concurrencia vales**: 10 `POST /vales` en paralelo → 10× 201, IDs `V-0013`–`V-0022` todos únicos, cero errores de PK.
- **Concurrencia ventas**: 6 `POST /ventas` en paralelo → 6× 201, `VT-0002`–`VT-0007` únicos.
- **Pagos por secuencia**: pago del registro aprobado del seed → `PG-0002` con monto congelado 87500 ✓; `PG-seed1` correctamente ignorado por el regex del `setval`.
- Código: `nextId(manager)` recibe el `EntityManager` de la transacción activa; el ID del vale se obtiene **dentro** de la misma transacción que inserta vale+tallas; `findLast()` y el `parseInt` sin guardia eliminados; `Date.now()` eliminado de pagos. SQL crudo limitado a los 3 `SELECT nextval(...)` estáticos (sin interpolación — cumple regla 5) — TypeORM no tiene API para secuencias.
- Migración bien defendida: regex `^V-[0-9]+$` contra IDs malformados (`PG-seed1`), `IF NOT EXISTS`, `down()` completo.
- **Hallazgo corregido en auditoría**: `setval(seq, N)` deja `is_called=true`, así que en BD vacía (`GREATEST(1,0)=1`) el primer `nextval` devolvía 2 → el primer vale de una instalación limpia habría sido `V-0002`. Corregido a `setval(seq, max+1, false)`; verificado desde cero: primer vale en BD vacía = **V-0001** ✓ (la migración estaba aplicada solo en entornos locales — amendarla era válido según la regla de inmutabilidad).
- Entorno dejado en estado limpio: BD recreada, migraciones + seed aplicados, script de prueba temporal eliminado.
