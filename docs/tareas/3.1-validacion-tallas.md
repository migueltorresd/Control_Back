# Tarea 3.1 — Validación estricta de `tallas`

> Fase 3 — Tests y CI · 🟠 Ítem 10 del plan · OWASP A03 · Depende de: fase 2 · Reglas: [README.md](README.md)

## Contexto / problema

`src/modules/vales/dto/create-vale.dto.ts:24-26` solo valida que `tallas` sea un objeto (`@IsObject`). Acepta valores negativos, decimales, no numéricos y claves no numéricas — y el controller hace `parseInt(tallaStr)` que produciría `NaN` como talla en BD.

## Archivos

- `src/modules/vales/dto/create-vale.dto.ts`
- `src/modules/vales/vales.controller.ts` (si la transformación cambia)

## Pasos

1. Implementar un validador custom (`@ValidatorConstraint`) para el campo `tallas` que exija:
   - Objeto no vacío.
   - Toda clave: entero entre 15 y 60 (rango razonable de tallas de calzado).
   - Todo valor: entero positivo.
2. Mensajes de error en español y específicos (qué clave/valor falló).
3. **No cambiar el shape del payload**: se mantiene `Record<string, number>` tal como lo envía el frontend hoy. No convertir a array anidado ni tocar `api.js` — esta tarea es solo validación.

## Criterio de aceptación

- [x] `POST /api/v1/vales` con `{"tallas": {"abc": -1}}` → 400 con mensaje claro en español.
- [x] `{"tallas": {}}` → 400. `{"tallas": {"40": 2.5}}` → 400. `{"tallas": {"40": 5}}` → 201.
- [x] El flujo normal del frontend (crear vale desde el modal) sigue funcionando.
- [x] Commit: `fix(vales): strict validation of tallas payload`.

## Registro de verificación

**2026-06-11 · Implementada y verificada ✅** (commit `ffa17af`)

- Casos probados en vivo (todos con su mensaje específico en español): `{"abc":-1}` → "La talla \"abc\" no es válida"; `{}` → "Debes indicar al menos una talla"; `{"40":2.5}` → "debe ser un número entero"; `{"99":5}` → "fuera del rango permitido (15 a 60)"; `{"40":-3}` → "debe ser mayor a 0"; `{"40":5}` → **201**.
- Shape `Record<string, number>` intacto — el frontend no necesitó cambios.
- **Hallazgo crítico colateral (corregido en `c245192`)**: al probar el caso válido, el vale nuevo recibió `V-0002` y **sobrescribió silenciosamente el vale del seed**. Dos causas: (1) el seed inserta IDs literales sin avanzar las secuencias (la migración las inicializa antes de que el seed corra); (2) `manager.save()` hace upsert ante colisión de PK. Fix: el seed ahora sincroniza las 3 secuencias (`setval max+1`) tras insertar, y las creaciones de vale/venta/pago usan `insert()` (falla ruidosa ante colisión, nunca sobrescribe). Re-verificado desde cero: secuencia en 3 tras el seed, vales nuevos `V-0003`/`V-0004`, seed intacto.
