# Tarea 3.1 — Validación estricta de `tallas`

> Fase 3 — Tests y CI · 🟠 Ítem 10 del plan · OWASP A03 · Depende de: fase 2 · Reglas: [README.md](README.md)

## Contexto / problema

`src/modules/vales/dto/create-vale.dto.ts:24-26` solo valida que `tallas` sea un objeto (`@IsObject`). Acepta valores negativos, decimales, no numéricos y claves no numéricas — y el controller hace `parseInt(tallaStr)` que produciría `NaN` como talla en BD.

## Archivos

- `src/modules/vales/dto/create-vale.dto.ts`
- `src/modules/vales/vales.controller.ts` (si la transformación cambia)

## Pasos

1. Implementar un validador custom (`@ValidatorConstraint`) para el campo `tallas` que exija:
   - Objeto no vacío.
   - Toda clave: entero entre 15 y 60 (rango razonable de tallas de calzado).
   - Todo valor: entero positivo.
2. Mensajes de error en español y específicos (qué clave/valor falló).
3. **No cambiar el shape del payload**: se mantiene `Record<string, number>` tal como lo envía el frontend hoy. No convertir a array anidado ni tocar `api.js` — esta tarea es solo validación.

## Criterio de aceptación

- [ ] `POST /api/v1/vales` con `{"tallas": {"abc": -1}}` → 400 con mensaje claro en español.
- [ ] `{"tallas": {}}` → 400. `{"tallas": {"40": 2.5}}` → 400. `{"tallas": {"40": 5}}` → 201.
- [ ] El flujo normal del frontend (crear vale desde el modal) sigue funcionando.
- [ ] Commit: `fix(vales): strict validation of tallas payload`.
