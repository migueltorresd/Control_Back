# Tarea 2.1 — Filtro de excepciones seguro

> Fase 2 — Seguridad perimetral · 🔴 Ítem 4 del plan · OWASP A05 · Depende de: fase 1 completa · Reglas: [README.md](README.md)

## Contexto / problema

`src/common/filters/http-exception.filter.ts:31-35` convierte **cualquier `Error` no-HTTP en 400 y expone `exception.message` al cliente**. Dos problemas:

1. **Fuga de información**: errores de TypeORM/pg (constraints, SQL fallido) revelan nombres de tablas/columnas y detalles internos.
2. **Enmascaramiento**: errores de programación reales aparecen como 400 "del usuario" en vez de 500, dificultando el diagnóstico.

## Archivos

- `src/common/filters/http-exception.filter.ts`

## Pasos

1. Mantener el formato de respuesta actual (`statusCode`, `timestamp`, `path`, `message`, `details`, `error`) — el frontend lo consume.
2. `HttpException` → responder su status y mensaje, como hoy.
3. **Cualquier otro error** → status 500, `message: 'Error interno del servidor'`, sin detalles del error original en la respuesta. Registrar con `Logger.error` (contexto `HttpExceptionFilter`) el mensaje real, el stack y la ruta.
4. Eliminar por completo el mapeo `Error → 400`.
5. Verificación previa: `grep -rn "throw new Error" src/` — los services ya usan excepciones de Nest (`BadRequestException`, `NotFoundException`...), así que no debería haber lógica de negocio dependiendo del mapeo viejo. Si aparece alguno, convertirlo a la excepción Nest correcta **en esta misma tarea**.

## Criterio de aceptación

- [x] Forzar un error de BD (p. ej. insertar por SQL directo una FK inexistente y consultar, o apagar la BD a mitad de request) → respuesta 500 genérica **sin** nombres de tablas/columnas; el detalle completo queda en el log del servidor.
- [x] Un 404 de negocio (`GET /vales/NO-EXISTE`) sigue mostrando su mensaje en español.
- [x] Las validaciones de DTO siguen devolviendo 400 con sus mensajes.
- [x] Commit: `fix(errors): generic 500 for unexpected errors, no internal details leaked`.
