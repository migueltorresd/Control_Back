# Tarea 3.3 — Test e2e del flujo completo

> Fase 3 — Tests y CI · 🟠 Ítem 9 del plan · Depende de: 3.2 · Reglas: [README.md](README.md)

## Contexto

Verificar el flujo de negocio completo contra una app real con BD real (no mocks): es lo que garantiza que las piezas (auth, transacciones, migraciones) funcionan juntas.

## Archivos

- `test/app.e2e-spec.ts` (reescribir el placeholder)
- `test/jest-e2e.json`
- Posible `test/utils/` para helpers (login, limpieza)

## Pasos

1. BD de test dedicada: base `control_produccion_test` en el mismo Postgres de compose. La config debe poder apuntar ahí vía env (`DATABASE_DATABASE=control_produccion_test` en el entorno del test).
2. En `beforeAll`: correr migraciones contra la BD de test (programáticamente con el DataSource de `src/config/data-source.ts`) y crear el usuario admin de test. En `afterAll` (o `beforeAll`): limpiar/recrear el esquema para que la suite sea **repetible**.
3. Flujo a cubrir (con `supertest`, todas las rutas bajo `/api/v1`):
   1. `POST /auth/login` con el admin → obtiene token (se usa en todo lo demás).
   2. `GET /vales` **sin token** → 401 (verificación fail-closed).
   3. Crear material/referencia/operario mínimos (o sembrar por repositorio) → `POST /vales` → 201 con ID `V-XXXX`.
   4. `POST /vales/:id/registro` (etapa Cortador, pares válidos) → 201, estado `registrado`.
   5. `PATCH /vales/:id/registro/:regId` a `aprobado` → monto congelado = pares × tarifa.
   6. `POST /pagos/:regId` → pago creado; el registro queda `pagado`.
   7. `POST /pagos/anular/:regId` → pago eliminado/anulado; el registro vuelve a `aprobado`.
   8. Intento de exceder el cupo del vale → 400.
4. Ajustar `test/jest-e2e.json` si hace falta (timeouts, setup).

- Flujo cubierto: 401 fail-closed → login admin de test (creado directo en BD con el hash de `AuthService`) → catálogo mínimo por API → vale `V-0001` por secuencia desde cero → tallas inválidas 400 → registrar → aprobar congela `4 × 1200 = 4800` → pagar (comprobante con monto congelado, registro `pagado`) → doble pago 400 → anular (vuelve a `aprobado`, cero comprobantes) → cupo excedido 400.
- Notas técnicas que quedan de plantilla: las migraciones se importan **explícitamente** (TypeORM no puede cargar globs de `.ts` bajo jest); `process.env.DATABASE_DATABASE` se fija **antes** de los imports del módulo; las respuestas de supertest se tipan con interfaces locales (`res.body as X`) para mantener lint en cero; el `DROP SCHEMA` también elimina la extensión `uuid-ossp` — funciona porque la migración inicial la crea explícitamente (fix de la auditoría 1.2).
