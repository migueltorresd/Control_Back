# Tarea 4.5 — Paginación de listados

> Fase 4 — Operabilidad · 🟡 Ítem 14 del plan · Depende de: fase 2 · Reglas: [README.md](README.md)

## Contexto

`GET /vales`, `GET /pagos` y `GET /ventas` cargan **todo** con relaciones, sin límite. Con meses de producción real, las respuestas crecerán sin control (memoria, latencia). Esta tarea cambia el shape de respuesta de las listas → **incluye la actualización del frontend** (regla 3 del README).

## Archivos

- Nuevo `src/common/dto/pagination-query.dto.ts`
- Repositories y controllers de `vales`, `pagos`, `ventas`
- **Frontend**: `src/services/api.js` y las vistas que consumen listas (`ValesView`, `PagosView`, `VentasView`, `HistorialView`, `DashboardView` — verificar cuáles listan)

## Pasos

1. `PaginationQueryDto` común: `page` (entero ≥1, default 1), `limit` (entero 1–200, default 50), opcionales `desde`/`hasta` (`@IsDateString`, filtro por campo `fecha`). Validado con `class-validator` + `@Type(() => Number)`.
2. Repositories: `findAndCount` con `skip`/`take` y filtros de fecha; conservar las relaciones y el orden actual (más reciente primero).
3. Respuesta estándar: `{ data: [...], total, page, limit }`. Los shapes de los elementos **no cambian** (siguen pasando por los mappers actuales).
4. Frontend, alcance exacto: (a) `api.js` acepta params opcionales de paginación y todas las vistas leen `data` del nuevo envelope; (b) controles de página (anterior/siguiente + indicador "página X de Y") **solo** en `HistorialView` y `PagosView`; las demás vistas usan el default (`limit=50`) sin controles. No rediseñar las vistas.
5. `GET /operarios`, `/materiales`, `/referencias` pueden quedar sin paginar (catálogos pequeños y acotados) — documentar esa decisión en el código del controller con un comentario solo si es necesario; no sobre-ingeniería.

## Criterio de aceptación

- [x] `GET /api/v1/vales?page=1&limit=10` responde N elementos y `total` correcto; `limit=999` → 400.
- [x] Filtro `?desde=...&hasta=...` funciona en pagos/ventas.
- [x] El frontend muestra todas las listas correctamente.
- [x] Commit: backend `bd332f5`, frontend `a385450`.

## Registro de verificación

**2026-06-12 · Implementada y verificada ✅** (backend `bd332f5`, frontend `a385450`)

**Decisión de diseño (consultada con el usuario): paginación opt-in.** El frontend calcula KPIs, liquidación y avance sobre el **conjunto completo** de datos en memoria ([useControlProduccion.js](../../../control-produccion/src/hooks/useControlProduccion.js)). Si los 3 endpoints pasaran a paginar por defecto (limit=50), esos cálculos quedarían incompletos y mostrarían cifras de dinero incorrectas **sin fallar visiblemente**. Para evitarlo:
- **Sin** params de query → la API devuelve el array completo (modo legacy). El hook de agregación sigue intacto.
- **Con** `?page`/`?limit`/`?desde`/`?hasta` → envelope `{ data, total, page, limit }`. Listo para consumidores futuros (storefront, móvil) y para listas largas.

**Verificado en vivo (curl):**
- `GET /vales` sin params → array de 3 (legacy) ✓
- `GET /vales?page=1&limit=1` → `{data:[1], total:3, page:1, limit:1}` ✓
- `GET /vales?limit=999` → 400 ✓
- `GET /pagos?desde=2026-06-01&hasta=2026-06-30` → envelope filtrado ✓
- `GET /pagos` sin params → array (legacy) ✓
- Integración: frontend 200, el hook recibe arrays (KPIs intactos).

**Backend:** `PaginationQueryDto` común (getter `esPaginado`, `limit` máx 200) + `findPaginated` (`findAndCount` con `Between`/`MoreThanOrEqual`/`LessThanOrEqual` sobre `fecha`) en los repos de vales/pagos/ventas. 19 e2e (4 nuevos de paginación), 41 unit, lint y build verdes.

**Frontend:** `api.js` — `getVales`/`getPagos`/`getVentas` aceptan `{ page, limit, desde, hasta }` opcionales (sin args → array completo). **Paginación de cliente** en `HistorialView` sobre los grupos por día (8 días/página, controles anterior/siguiente + "página X de Y", visibles solo con >1 página), porque el hook ya carga todos los pagos para la liquidación — no tiene sentido una segunda llamada al servidor. Los KPIs del Historial siguen calculados sobre el total.

**Decisión documentada:** `PagosView` lista operarios (~10, acotado) y `/operarios`, `/materiales`, `/referencias` son catálogos pequeños — no se paginan, como anticipaba el paso 5 de la tarea.
