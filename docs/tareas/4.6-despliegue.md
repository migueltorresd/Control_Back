# Tarea 4.6 — Despliegue reproducible con TLS

> Fase 4 — Operabilidad · 🟡 Ítem 15 del plan · OWASP A02/A05 · Depende de: 4.4 (health para depends_on) · Reglas: [README.md](README.md)

## Contexto

Hoy `compose.yaml` solo levanta la BD, con `postgres/postgres` hardcodeado y el puerto 5432 expuesto al host. No hay imagen del backend ni procedimiento de despliegue documentado. **TLS es obligatorio en producción**, no opcional (OWASP A02): el token JWT y las credenciales viajan en cada request.

## Archivos

- `Dockerfile` (nuevo, raíz del backend) + `.dockerignore`
- `compose.yaml` (extender)
- `README.md` (sección Despliegue + Backups)

## Pasos

1. `Dockerfile` multi-stage:
   - Stage deps/build: `node:22-alpine`, pnpm via corepack, `pnpm install --frozen-lockfile`, `pnpm build`.
   - Stage runtime: `node:22-alpine`, solo `dist/` + `node_modules` de producción (`pnpm install --prod`), usuario **no root** (`USER node`), `CMD ["node", "dist/main"]`.
   - `.dockerignore`: `node_modules`, `dist`, `.env`, `.git`.
2. `compose.yaml`:
   - Servicio `db`: credenciales desde `.env` (`${DATABASE_USERNAME}` etc., nada hardcodeado), **sin** exponer 5432 (o solo `127.0.0.1:5432:5432` para administración local), `healthcheck` con `pg_isready`.
   - Servicio `backend`: build del Dockerfile, `env_file: .env`, `depends_on: db: condition: service_healthy`, expone el puerto solo a la red interna del compose (el proxy es quien publica).
3. **Reverse proxy con TLS (único punto expuesto)**: añadir servicio `caddy` con `Caddyfile` que sirva el frontend estático (build de Vite) y haga proxy de `/api/*` al backend. **Usar Caddy** (TLS automático con Let's Encrypt) — no nginx, salvo que el cliente ya opere nginx, en cuyo caso detenerse y consultar (regla 8 del README). En desarrollo local el proxy no se usa; el compose debe permitir levantar solo db+backend.
4. README — sección "Despliegue" con los pasos exactos en orden:
   1. Configurar `.env` (desde `.env.example`, credenciales fuertes).
   2. `docker compose up -d`.
   3. `pnpm migration:run` (o ejecutarlo dentro del contenedor).
   4. `pnpm create-admin`.
   5. Verificar `GET /api/v1/health`.
5. README — sección "Backups": `pg_dump` programado (tarea del SO o servicio en compose), retención sugerida (diario × 14 + mensual), y cómo restaurar (`psql < dump`). Probar una restauración una vez.

## Registro de verificación

**2026-06-12 · Implementada · verificación parcial ✅⚠️** (backend `3de2809`, frontend `ff6a755`)

**Decisión de arquitectura (consultada): despliegue en nube con dominio.** El stack es `db + backend + caddy`; Caddy es el único expuesto (80/443) y obtiene TLS automático de Let's Encrypt. Frontend y `/api/*` van por el mismo dominio (mismo origen, sin CORS cross-origin).

**Verificado en vivo:**
- **`migration:run:prod`** con el data-source **compilado** (`dist/config/data-source.js`) → corrió todas las migraciones contra la BD. Esto valida lo más riesgoso del despliegue: las migraciones corren desde `dist/` dentro del contenedor, sin ts-node ni código fuente.
- **`create-admin:prod`** (`node dist/seed/create-admin.js`) → admin creado y confirmado en BD.
- **Backup/restore**: `pg_dump` → 859 líneas / 14 tablas; restaurado en BD nueva → 14 tablas + admin presentes.
- **`docker compose config`**: interpola bien; `DATABASE_HOST` forzado a `db`; `db` solo en `127.0.0.1:5432`; backend con `expose` (sin puerto al host); Caddy publica 80/443.
- **`pnpm build`** del backend (nest) y del frontend (vite) verdes; data-source dual no rompió dev (`migration:run` dev OK, 41 unit tests verdes); lint del data-source limpio.

**No verificado por el entorno (no por el código):** este Docker local no pudo construir las imágenes — buildkit falla por permisos de cgroups y el daemon no autentica el pull de `node:22-alpine`/`caddy:2-alpine`. Por tanto **quedó sin ejecutar**: el build de las imágenes, el levantamiento del stack completo con Caddy/TLS, `whoami` → node en el contenedor y la prueba del frontend servido por Caddy. La configuración está validada sintácticamente y el Dockerfile sigue el patrón estándar (multi-stage, `USER node`). **Debe probarse en el servidor real** (o un Docker funcional) con `docker compose up -d --build` antes de la entrega — es el paso final del UAT de despliegue.

**Hallazgo corregido (commit `69955e2`):** al correr `pnpm seed` durante la verificación, fallaba con *"Entity metadata for Vale#rechazos was not found"* — bug preexistente de la tarea 6.2: la entity `Rechazo` y la relación `Vale.rechazos` se agregaron, pero `Rechazo` no se incluyó en la lista de entities del `SeedModule`. Añadida; seed verificado OK.
