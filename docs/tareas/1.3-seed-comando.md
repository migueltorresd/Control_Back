# Tarea 1.3 — Seed como comando manual

> Fase 1 — Integridad de datos · 🔴 Ítem 3 del plan · Depende de: 1.2 · Reglas: [README.md](README.md)

## Contexto / problema

`src/services/seed.service.ts` implementa `OnModuleInit`: en **cada arranque** de la app inserta datos de demostración (operarios ficticios, vales, pagos) si la BD está vacía. En la BD del cliente esto contaminaría datos reales tras una instalación limpia o un reset. El arranque de una app no debe mutar datos (patrón "seeder como comando", grupo B del plan maestro).

Nota: `SeedService` es además la única pieza que inyecta `Repository<T>` de TypeORM directo (vía el `TypeOrmModule.forFeature` de `AppModule`), violando la convención repository-only. Esta tarea elimina esa excepción.

## Archivos

- `src/seed/seed.module.ts` y `src/seed/seed.ts` (nuevos)
- `src/services/seed.service.ts` (mover a `src/seed/seed.service.ts`)
- `src/app.module.ts`
- `package.json`

## Pasos

1. Quitar `implements OnModuleInit` y el método `onModuleInit` del seed service.
2. Sacar `SeedService` y el `TypeOrmModule.forFeature([...])` de `AppModule`.
3. Crear `SeedModule` propio (importa `ConfigModule`, `TypeOrmModule.forRoot...` reutilizando la config de la tarea 1.1, y el `forFeature` con las entidades que el seed necesita) y mover allí el service.
4. Crear `src/seed/seed.ts`: script standalone con `NestFactory.createApplicationContext(SeedModule)` → ejecuta `seed()` → `app.close()`. Mantener la idempotencia actual (si hay datos, loguea "ya hay datos, no se siembra" y sale con código 0).
5. Script en `package.json`: `"seed": "ts-node -r tsconfig-paths/register src/seed/seed.ts"`.
6. Verificar que las fechas/datos del seed siguen siendo coherentes (son datos demo, está bien).

## Criterio de aceptación

- [x] La app arranca sin sembrar nada (BD vacía permanece vacía).
- [x] `pnpm seed` puebla una BD vacía; ejecutado de nuevo, no duplica y lo informa.
- [x] Commit: `refactor(seed): manual seed command, removed from app bootstrap`.
